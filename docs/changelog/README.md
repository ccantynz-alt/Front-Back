# Crontech Changelog System

`scripts/generate-changelog.ts` is the AI-authored release-notes pipeline.
On every push to `Main`, the script reads the commits since the most
recent boundary marker in `CHANGELOG.md`, asks Claude Haiku to draft
three sections, and writes them to disk.

## What it produces

Per run, the generator writes three artifacts:

1. `CHANGELOG.md` — the new entry is **prepended above** the previous
   boundary marker so the file reads newest-first. A fresh
   `<!-- changelog-boundary:YYYY-MM-DD -->` marker closes the entry so
   the next run knows where to pick up.
2. `docs/changelog/<date>.md` — the same entry as a standalone file,
   useful for linking to a single release from the docs site.
3. `docs/changelog/<date>.json` — structured payload for the admin UI:

   ```json
   {
     "date": "2026-04-19",
     "fromBoundary": "2026-04-12",
     "toSha": "abcdef0...",
     "commitCount": 7,
     "commits": [{ "sha": "...", "shortSha": "...", "subject": "...", "author": "...", "date": "..." }],
     "sections": {
       "userNotes": "- ...",
       "devNotes": "...",
       "social": "..."
     }
   }
   ```

## Three sections

Each release has three Claude-authored sections, generated by three
separate Haiku calls so each prompt stays sharp and short:

| Section | Audience | Tone | Length |
|---|---|---|---|
| User notes | End users | Polite, marketing-safe, no competitor names, what users can now DO | 3-6 bullets |
| Dev notes | Engineers | Plain engineering English, breaking changes, migrations | As needed |
| Social snippet | Public timelines | Direct, no buzzwords, no gratuitous emojis | < 280 chars |

Every reply is validated with Zod before it is written. An over-length
social snippet, an empty user-notes block, or a missing field fails
fast — the run aborts and nothing is written.

## Boundary markers

Each release is closed by a marker:

```
<!-- changelog-boundary:YYYY-MM-DD -->
```

These markers are the source of truth for "where did we get up to last
time?". Do not edit them by hand. The generator inserts new markers and
keys off the most recent one when computing `git log <date>..HEAD`.

If `CHANGELOG.md` is missing entirely, the generator seeds it with a
header explanation plus today's marker and proceeds as if it just
shipped its first release.

## CLI

```
bun run scripts/generate-changelog.ts [--from=<sha>|<YYYY-MM-DD>] [--dry-run]
```

- `--from=<value>` — override the boundary lookup. Accepts a sha or a
  YYYY-MM-DD date.
- `--dry-run` — read commits, print what WOULD be sent to Claude and
  written to disk, but make no Claude calls and write no files. Useful
  in CI to verify the script is healthy without burning tokens.

If `ANTHROPIC_API_KEY` is not set in the environment, the script falls
back to the same preview behaviour as `--dry-run` so it never errors
on a fresh checkout.

## Testing

`scripts/generate-changelog.test.ts` mocks the Claude client and
exercises every code path: arg parsing, boundary detection, git-log
parsing, prompt assembly, Zod validation (including over-length social
replies), markdown rendering, and the prepend-above-latest-boundary
insertion logic.

```
bun test scripts/generate-changelog.test.ts
```

The tests never touch the network and never write to the real
filesystem — `runGeneration()` is a pure function of its inputs and
returns the would-be file contents for the caller to assert against.
